# بروتوكول التكامل السيادي مع منصة Gemini Enterprise Agent Platform

توثق هذه الورقة معمارية الربط والتكامل الفائق بين نظام التشغيل السيادي **Nexum-Core** وحزمة خدمات **Gemini Enterprise Agent Platform** من جوجل، لتوليد بيئة هجينة تجمع بين سيادة التحكم المحلي وكفاءة وقابلية التوسع السحابي اللانهائي.

## 1. التنسيق بين الوكلاء (Agent-to-Agent Orchestration)
* **الربط السيادي:** يتكامل مع بروتوكول `InterBotProtocol` ومحرك `YAMLProtocolEngine` لـ Nexum.
* **الآلية:**
  - يتم إدراج الوكلاء السحابيين المدارين من جوجل (الذين تم بناؤهم بـ Dialogflow CX) كوكلاء تنفيذ فرعيين (Sub-Agents) داخل سجل الكيان لنيكسوم.
  - عند قيام محرك الـ DAG بتشغيل بروتوكول صيانة معقد، يتم تفويض الجزء السحابي (مثل ضبط أوزان الـ Load Balancer أو توسيع عناقيد GKE) إلى `GoogleAgentPlatformConnector` الذي يستدعي بدوره الوكيل السحابي المخصص لإنهاء المهمة وإرجاع النتيجة مشفرة.

---

## 2. الهوية التشفيرية والوصول المدار (Agent Identity & IAM)
* **الربط السيادي:** يعتمد على توقيعات `AuditEvent` بنظام HMAC-SHA256 المحلي في نيكسوم.
* **الآلية:**
  - يمنح نيكسوم كل وكيل (مثل `Sentinel` أو `GithubAgent`) مفتاح تشفير محلي فريد مشتق من `SOVEREIGN_HMAC_KEY`.
  - يتم ربط هذا التوقيع بـ **Google Cloud IAM Service Accounts** من خلال بوابة الصلاحيات المؤقتة.
  - عند طلب أي تنفيذ سحابي، يتواصل `SovereignExecutionGateway` مع خدمة GCP STS (Security Token Service) لتوليد **مفتاح دخول مؤقت (Short-lived OAuth Access Token)** صالح لـ 15 دقيقة فقط ومقيد بمهام الأداة المستدعاة، مطبقاً نظرية *انعدام الثقة الكامل (Zero Trust Architecture)*.

---

## 3. بوابة الوكلاء (Agent Gateway)
* **الربط السيادي:** تندمج مع بوابة العبور السيادية لنيكسوم `SovereignExecutionGateway`.
* **الآلية:**
  - تعمل بوابة نيكسوم كـ "فلتر سيادي محلي" يفحص المدخلات والمخرجات ضد عمليات الحقن غير المباشر (Prompt Injection) ومحاولات الخروج عن المسار.
  - تمرر البوابة الطلبات النظيفة والموقعة إلى **Google Agent Gateway** السحابية التي تدير حركة المرور وتنسق الاتصال مع الأنظمة الخلفية السحابية والبيانات المشتركة بأعلى معايير الحماية والتوافقية.

---

## 4. خوادم MCP المدارة بالكامل (Google Managed MCP Servers)
* **الربط السيادي:** تترابط مع `gcp_mcp.py` و `mcp_dispatcher.py` لنيكسوم.
* **الآلية:**
  - بدلاً من استدعاء مكاتب APIs برمجية معقدة أو الحفاظ على خوادم MCP محلية ثقيلة، يتحول نيكسوم لاستخدام **Google Managed MCP Servers**.
  - يتم تبسيط الاستعلامات لخدمات مثل Compute Engine أو Cloud Storage أو BigQuery لتتم عبر لغة طبيعية مباشرة يمررها نيكسوم للـ Managed remote MCP Endpoints التابعة لجوجل، محققاً فلسفة **"بنية تحتية صفرية للربط بالخدمات" (Zero Infrastructure Integration)**.

---

## 5. بيئات التنفيذ الآمنة السحابية (Secure Sandboxes)
* **الربط السيادي:** توسيع للمتصفح الآمن ومحاكي الأكواد المحلي لنيكسوم `browser_sandbox.py`.
* **الآلية:**
  - يتبنى نيكسوم معمارية التنفيذ الهجين:
    - **التنفيذ المحلي (Local Sandbox):** للمهام البسيطة والمعلومات الحساسة التي تتطلب حماية تامة من التوصيل الخارجي.
    - **التنفيذ السحابي (Cloud Secure Sandbox):** للمهام الكبيرة، تصفح المواقع غير الموثوقة عبر الويب (Web Automation / Computer Use)، واختبار الأكواد البرمجية المولدة آلياً.
  - يتم إرسال الكود أو رابط التصفح للـ Secure Cloud Sandbox المدار من جوجل، وتلقي النتائج واللقطات الموقعة رقمياً، لحماية خادم نيكسوم المحلي بالكامل من أي ثغرات أو استهلاك مفرط للموارد.

---

## 6. بنوك الذاكرة المستمرة المهجنة (Hybrid Memory Architecture)
* **الربط السيادي:** دمج مباشر مع `SovereignMemoryStore` و `MemoryManager` في نيكسوم.
* **الآلية:**
  - ينقسم بنك الذاكرة إلى مستويين متكاملين:
    - **المستوى الأول (Local Private Memory):** محلي بالكامل على خادم المايسترو (SQLite FTS5)، ويحتوي على مفاتيح التشفير، التفضيلات السيادية، وذاكرة الأخطاء والترميم الذاتي الحساسة.
    - **المستوى الثاني (Shared Cloud Memory Bank):** سحابي مدار بالكامل، يتم تحديثه تلقائياً بالوثائق الفنية الضخمة، المرجعيات، ومحاضر الجلسات التنفيذية الطويلة.
  - يتحرك الـ `MemoryManager` بذكاء بين الذاكرتين لضمان استرجاع سياقي دقيق دون تضخيم سياق المحادثة المحلي أو المساس بسرية البيانات.

---

## 7. محاكاة وتقييم القرارات قبل التنفيذ (Simulation & Evaluation)
* **الربط السيادي:** يعمل كخط تحقق علمي قبل قرارات `CouncilDebateProtocol` (مجلس الحكماء).
* **الآلية:**
  - عند ابتكار الحارس (Sentinel) لكود ترميم ذاتي (Self-Healing Patch) معقد للنظام، لا يتم تطبيقه مباشرة.
  - يرسل نيكسوم الكود المقترح مع سيناريو التشغيل إلى **Google Agent Simulation Suite** ليتم اختباره ضد آلاف التفاعلات والطلبات الاصطناعية (Synthetic Traffic).
  - يتم تجميع تقارير المحاكاة ومعدل الخطأ (Evaluation Drift Metrics) وتمريرها فوراً لمجلس الحكماء (3 Sages). إذا حصل الكود على تقييم نجاح كافٍ، يصدر المجلس قراره بالإجماع ويتم النشر على PM2 بأمان تام.

---

## 8. السحابة المتوافقة مع تفكير الوكلاء (Agentic Data Cloud)
* **الربط السيادي:** يسرع من عمليات الفهرسة المحلية المعرضة للتوقف `index_full.py` و `index_codebase.py`.
* **الآلية:**
  - يتم ترحيل الفهرسة المعرفية الدلالية والبحث المتجهي لقواعد المعرفة الضخمة لتعمل مباشرة داخل قاعدة البيانات السحابية (مثل AlloyDB أو BigQuery مع دعم Vector Search مدمج).
  - يستدعي نيكسوم محرك البحث المعرفي السحابي عبر موصلات `DiscoveryEngineConnector` المتاحة، لتقديم نتائج RAG فائقة السرعة وفي أجزاء من الثانية دون إرهاق معالج السيرفر المحلي.
